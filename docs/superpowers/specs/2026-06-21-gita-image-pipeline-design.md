# Gita Kids — Image Generation Pipeline Redesign

**Date:** 2026-06-21
**Status:** Design approved (pending spec review) — pilot on Chapter 1
**Related:** `docs/illustration-guidelines.md`, `scripts/generate-illustration.mjs`, `docs/MOBILE-APPS.md`

---

## 1. Problem

Current illustrations (generated by `scripts/generate-illustration.mjs`) have three recurring defects, traced to the pipeline:

1. **Off-topic** — the only "scene" sent to the model is the verse's abstract `meaning` text plus the story title. Abstract philosophy in → loosely-related picture out.
2. **Inconsistent characters** — characters are described in **text only**; **no reference image** is sent, so the model redraws each character from scratch every time (a character "looks different in one sentence and the next"). Overlapping descriptions also make distinct characters look alike.
3. **Extra / wrong figures** — the cast is guessed by keyword-matching the meaning text, and the "fill all space" (horror vacui) rule invites invented figures. No QA verifies the result against the verse.

## 2. Goals / non-goals

**Goals**
- Characters look identical and distinct across all images in a chapter.
- Each image connects to its verse at the right level (see §3).
- No stray figures; no text rendered in images.
- A repeatable, **scalable** system that templates from the Chapter 1 pilot to all 18 chapters.

**Non-goals (this iteration)**
- No change to the folk-art **styling** (the user is happy with it).
- No image optimization / app bundling (tracked separately in `docs/MOBILE-APPS.md`).
- No hi-res-originals datastore (that's roadmap item 2; this pipeline outputs the web/app image).

## 3. Governing principle — "specificity on demand"

Each verse's brief carries a **`relevance_mode`**:

| Mode | When | Bar the image must meet |
|------|------|-------------------------|
| `specific` | verse names a concrete, depictable moment (a named warrior blows a named conch; Arjuna collapses in the chariot) | depicts that moment accurately, correct cast |
| `evocative` | abstract/philosophical verse | a symbolic image that **connects to the theme**; creative liberty allowed; **total disconnect fails** |
| `devotional` | devotional chapters (e.g. Bhakti Yoga) | a beautiful on-theme image (Krishna, gopis, lotus ponds) is fine without line-specificity |

Chapter 1 (the battlefield muster) is mostly `specific`, which makes it a clean pilot.

## 4. Scope & roster

- **Pilot:** Chapter 1 — Arjuna Viṣāda Yoga, all 47 verses, **Madhubani** style. Regenerate **all 47** (mixing old text-only images with new ref-conditioned ones would look inconsistent).
- **Roster (user decision):** locked model sheets for the **core 7** — `dhritarashtra, sanjaya, duryodhana, bhishma, drona, arjuna, krishna`. Minor named figures in the muster verses (Karna, Kripa, Bhima, the conch-blowers, etc.) are drawn from **generic warrior archetypes** (e.g. a "Kaurava warrior" group and a "Pandava warrior" group, distinguished by armor palette), not individual sheets.

## 5. Components

### 5.1 Scene briefs
- **File:** `content/chapters/{slug}/scene-briefs.yaml` — one entry per verse:
  ```yaml
  - verse: 1
    relevance_mode: specific        # specific | evocative | devotional
    cast: [dhritarashtra, sanjaya]  # core-7 keys and/or archetype keys
    action: "Blind king Dhritarashtra on his throne asks Sanjaya what happened on the field"
    setting: "Hastinapura palace hall; battlefield glimpsed through an arch"
    props: ["throne", "blindfold"]
    notes: ""
  ```
- **Authoring:** parallel agents read each verse's `meaning` + `story` and draft a brief, choosing the `relevance_mode` (specific / evocative / devotional) per verse.
- **Review gate (REQUIRED):** the user approves/edits the 47-brief list **before any image is generated**. This is where relevance is locked, cheaply, in text.

### 5.2 Character model sheets (reference images)
- **Core 7**, one canonical portrait each, in the chapter's folk style, on a plain patterned field, showing the locked attributes from `docs/illustration-guidelines.md` §1. Deliberately differentiated (age, palette, headwear, weapon) so they never look alike.
- **Archetypes:** 2–3 generic warrior references (Kaurava / Pandava), used for minor named figures and crowds.
- **Per style:** a character needs one sheet **per folk style** it appears in (Arjuna-in-Madhubani for Ch 1; Arjuna-in-Warli later). The pilot produces Madhubani sheets only.
- **Storage:** `assets/character-refs/{style}/{key}.png` (source inputs — NOT under `public/`, not served).
- **Locking:** generate 2–3 candidates per character → user picks the canonical one → frozen. Approval gate.

### 5.3 Generation pipeline
Extend `scripts/generate-illustration.mjs` (or a sibling `generate-v2.mjs`) to:
1. Load the verse, its **scene brief**, and the chapter style.
2. Resolve `cast` → load the relevant **ref image(s)** from `assets/character-refs/{style}/`.
3. Build the prompt from the **brief** (action/setting/props) — not the abstract meaning — plus the style block, palette, and an explicit cast constraint: *"Depict exactly these figures: … No other people. No text."*
4. Send a **multi-part** request to Gemini (image model): `[refImagePart…, { text: prompt }]`. (The current call sends text only — this is the core fix.)
5. `evocative` / `devotional` modes relax the strict-cast clause and emphasize mood/theme.
6. Write to `public/illustrations/{slug}/{verse}.png`.

### 5.4 QA loop
- After generation, a **vision check** sends the produced image + brief to a multimodal model and requests a structured verdict:
  ```json
  { "cast_correct": true, "action_present": true, "no_extra_people": true,
    "no_text": true, "style_ok": true, "overall_pass": true, "issues": [] }
  ```
- For `specific`: all checks apply. For `evocative`/`devotional`: replace cast/action checks with `connected_to_theme` + `style_ok` + `no_text`.
- **On fail:** regenerate with the `issues` appended as corrective feedback. Max **3 attempts**; if still failing, flag for manual review.
- **Log:** write a per-chapter QA report (verse, attempts, final verdict, issues) for the user to scan.

### 5.5 Staged approval gates
1. **Briefs** (47) → user approves.
2. **Model sheets** (core 7 + archetypes) → user approves the canonical set.
3. **Sample** — generate 5 representative verses → user approves the look.
4. **Full chapter** — generate remaining verses + QA → user reviews the report.

## 6. Data flow

```
verse.yaml (meaning + story)
   └─(agent)→ scene-briefs.yaml ──[USER APPROVES]
                                     │
character sheet (guidelines §1) ─(gen)→ assets/character-refs/{style}/*.png ──[USER APPROVES]
                                     │
   brief + ref images ─(gen, multi-part)→ candidate PNG
                                     │
   candidate + brief ─(QA vision)→ pass? ──no──> regenerate (≤3)
                                     │yes
                          public/illustrations/{slug}/{verse}.png  + QA report
```

## 7. Edge cases
- **Muster verses with many warriors** → `cast` uses archetype keys; prompt says "a row of Pandava/Kaurava warriors," no individual identity required.
- **Pure framing verses** (Dhritarashtra/Sanjaya) → core-7 refs, `specific`.
- **Model returns no image / rate limit** → existing retry/fallback logic in the script is kept.
- **QA fails 3×** → keep the best attempt, flag in the report; do not silently ship.
- **Regeneration safety** → current Ch 1 images are in git history; the run overwrites in place and is revertible.

## 8. Scaling to all 18 chapters
- The brief + generation + QA system is **style-agnostic** and reused as-is.
- Per chapter: author briefs, generate the **model sheets in that chapter's folk style** for whichever core-7 members appear, then run generation + QA.
- Character attributes stay fixed (guidelines §1); only the style of the sheet changes.

## 9. Verification
- Pilot success = Chapter 1's 47 images: (a) core-7 characters visually consistent across the chapter, (b) each image meets its `relevance_mode` bar, (c) no stray figures, (d) no text, (e) QA report all-pass or manually-cleared.
- Spot-check on the live site (Remote WebView app reflects them automatically).
